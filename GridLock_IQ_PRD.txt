1. Executive Summary
Bengaluru's traffic enforcement is fundamentally reactive. Officers patrol based on experience, not data. When illegal parking chokes an intersection at Koramangala at 6 PM on a Saturday, no alert fires. No officer is pre-positioned. The congestion compounds until someone physically responds.

GridLock IQ flips this entirely.

Operating strictly within the zero-external-dataset constraint, GridLock IQ derives a physics-inspired Accumulated Obstruction Index (AOI) entirely from internal violation logs using three key dataset columns most teams will ignore: vehicle_type (physical mass), closed_date minus created_date (obstruction duration), and spatial density within Uber H3 hexagonal cells.

The system predicts gridlock 1 to 4 hours in advance and auto-generates patrol dispatch instructions naming specific police stations, zones, and recommended unit counts. Beyond prediction, it includes a what-if enforcement simulator that lets officers compare current patrol patterns against AI-optimized deployment — quantifying the expected congestion reduction in percentage terms.

2. Problem Statement
2.1 Operational Challenge
On-street illegal parking and spillover parking near commercial areas, metro stations, and events actively choke carriageways and intersections across Bengaluru. The problem is not the existence of violators — it is the inability to anticipate and respond before congestion cascades.

Challenge 1
Enforcement is patrol-based and reactive. Officers respond after gridlock forms, not before.
Challenge 2

Challenge 3

Challenge 4

Challenge 5

2.2 Problem Statement Direction

3. Core Technical Innovation — The AOI
3.1 The Zero-External-Dataset Constraint
Competition rules explicitly prohibit external datasets for Theme 1. Teams using Google Maps API, TomTom speed logs, or any supplementary traffic data face immediate disqualification. GridLock IQ fully complies by deriving all congestion signals from the violation log itself.

3.2 The Hidden Dataset Columns
Most teams will use only: latitude, longitude, vehicle_type, and created_date. GridLock IQ exploits three additional columns that most teams will ignore:

closed_date
Minus created_date gives exact obstruction duration in minutes — a direct proxy for how long a lane was blocked.
junction_name
Violations at junctions have 2x the congestion impact of mid-block violations. This is a free multiplier.
validation_status
Approved vs rejected tells you enforcement ROI per zone — which patrols are effective and which are wasted.

3.3 Accumulated Obstruction Index (AOI) — The Core Formula
AOI is a physics-inspired metric that simulates lane capacity degradation using purely internal features:

AOI  =  VehicleMass  x  ObstructionDuration  x  LocalDensity  x  JunctionMultiplier

Where:
  VehicleMass         =  TANKER=5.0  |  BUS=4.5  |  CAR/PASSENGER=3.0
                         TAXI=2.5  |  MOTOR CYCLE=1.5  |  SCOOTER=1.0

  ObstructionDuration =  closed_date - created_date  (in minutes)
                         If closed_date is NULL: use rolling mean of that H3 cell

  LocalDensity        =  count of concurrent open violations in same H3 hex cell
                         within a rolling 3-hour temporal window

  JunctionMultiplier  =  2.0 if junction_name is not null, else 1.0

  Final AOI normalised to 0-100 scale:
    0-25   =  Low Risk (Green)
    26-50  =  Moderate Risk (Yellow)
    51-75  =  High Risk (Orange)
    76-100 =  Critical Gridlock (Red)

3.4 Why This Is Defensible
When judges challenge 'how are you measuring congestion without traffic data', the answer is:

A TANKER parked at a junction for 45 minutes blocks more lanes than a scooter parked for 5 minutes — this is physics, not assumption.
Multiple concurrent violations in the same 100-metre cell create compounding obstruction — density is real congestion signal.
Junction violations have measured higher congestion impact in urban traffic literature — the 2x multiplier is conservative.
The normalization to 0-100 allows intuitive communication with traffic police — they understand risk levels, not raw numbers.

4. Feature Specifications
GridLock IQ is built around 8 features, prioritized into three tiers based on top-10 differentiation impact.

F1: Uber H3 Spatial Binning + HDBSCAN Clustering
MUST BUILD

Purpose
Map individual GPS coordinates into discrete 100-metre hexagonal cells using Uber's H3 spatial index, then run HDBSCAN to discover corridor-level hotspot clusters.

Why H3 over standard grid or radius-based binning
Hexagonal cells eliminate edge artifacts that square grids produce at cell borders.
H3 resolution level 9 produces 100-metre cells — correct scale for street-level parking analysis.
H3 IDs are stable, hashable keys — easy to aggregate across time windows without floating-point errors.
3D rendering of H3 cells as hex pillars (via PyDeck) is visually distinctive and immediately interpretable.

Why HDBSCAN over DBSCAN
Bengaluru's violation density is non-uniform — commercial corridors (Koramangala, MG Road) are dense, residential areas sparse. HDBSCAN handles variable density natively; DBSCAN does not.
HDBSCAN outputs a confidence score per cluster, enabling soft boundary detection — useful for flagging 'emerging hotspots'.

Technical Specification
Input
Cleaned dataframe with latitude, longitude, computed AOI score, timestamp
Library
uber-h3-py for binning, hdbscan for clustering
H3 Resolution
Resolution 9 (average hexagon area: 0.105 km²)
Temporal Window
Rolling 3-hour aggregation per H3 cell
Output
H3 cell ID → aggregated AOI score, cluster label, violation count, dominant vehicle type
Saved As
h3_clustered_zones.parquet

F2: Multi-Horizon LightGBM Forecasting Engine
MUST BUILD

Purpose
Predict the AOI score for every active H3 cell at T+1h, T+2h, and T+4h horizons. This shifts GridLock IQ from a reporting tool to a predictive intelligence system.

Feature Engineering for Forecasting
Lag features: AOI at T-1h, T-2h, T-3h per H3 cell
Cyclic time encoding: sin(2π × hour/24), cos(2π × hour/24), sin(2π × day/7), cos(2π × day/7) — prevents the model treating 23:00 and 00:00 as maximally different
Rolling statistics: 7-day rolling mean AOI, 7-day rolling max AOI, 7-day rolling std per cell
Junction flag: binary indicator for junction-adjacent H3 cells
Vehicle type mode: most frequent vehicle type in the H3 cell historically
Historical density: average concurrent violations per cell at same hour across all historical data

Model Training Strategy
Algorithm
LightGBM Regressor (fast, handles non-linear temporal patterns well)
Train/Test Split
Chronological — first 80% of timeline for training, last 20% for validation. Never shuffle time series.
Three Models
Separate model for T+1h, T+2h, T+4h prediction horizons
Validation Metric
MAE (Mean Absolute Error on AOI scale 0-100), RMSE, R²
Serialization
joblib.dump() → /models/lgbm_t1.pkl, lgbm_t2.pkl, lgbm_t4.pkl
Expected MAE
Target < 8 points on 0-100 AOI scale for T+1h horizon

F4: Automated Dispatch Alert Engine
MUST BUILD

Purpose
Auto-generate plain-text operational dispatch instructions when predicted AOI exceeds 75 (Critical threshold) in any H3 cell.

Alert Logic
for each H3 cell in forecast_t1h:
    if predicted_AOI >= 75:
        station = reverse_lookup(h3_cell, center_code_jurisdiction_map)
        vehicle_type = dominant_vehicle_type(h3_cell)
        units = ceil(predicted_AOI / 25)  # 1-4 units based on severity
        generate_alert(
            f'CRITICAL ALERT: Predicted gridlock at {zone_name} in 60 minutes.'
            f'Severity: {predicted_AOI:.0f}/100  |  Primary violator: {vehicle_type}'
            f'Recommended: Deploy {units} towing unit(s) from {station}.'
            f'Time window: {current_time} to {current_time + 2h}'
        )

Sample Alert Output
CRITICAL ALERT — Koramangala 5th Block
Predicted AOI: 88/100  |  Horizon: 60 minutes  |  Primary violator: CAR
Action: Deploy 3 towing units from BTP051 - Shivajinagar immediately.

F5: Repeat Offender Intelligence
STRONG

Purpose
Track vehicle_number across violation history to identify chronic violators — vehicles that accumulate multiple violations across multiple zones. These become targeted enforcement priority targets.

Scoring Formula
Repeat_Risk_Score = (violation_count * 0.5)
                  + (unique_locations_count * 0.3)
                  + (recency_weight * 0.2)

Where recency_weight = violations in last 30 days / total violations

Risk Tier:
  Score >= 10  =  HIGH PRIORITY (immediate towing on next detection)
  Score 5-9    =  MEDIUM PRIORITY (flagged for checkpoint stops)
  Score < 5    =  MONITOR

Output
Top 50 repeat offenders ranked by risk score
Per-vehicle violation map showing all locations
Integration with dispatch alerts: if a HIGH PRIORITY vehicle is detected in a violation, alert is upgraded

F7: Enforcement ROI Score
STRONG

Purpose
This feature is original — neither reference PRD includes it. The validation_status column (approved/rejected) reveals which enforcement actions actually resulted in cleared violations. Using this to compute per-zone enforcement efficiency exposes where patrol effort is wasted.

Formula
Enforcement_ROI(zone) = approved_violations / (approved + rejected) * 100

Interpretation:
  ROI > 80%   =  High efficiency zone — patrol here is effective
  ROI 50-80%  =  Medium efficiency — patrol with targeted focus
  ROI < 50%   =  Low efficiency — either false positives or structural issue
                  Recommend: infrastructure intervention over patrol

Actionable Output
Per-zone ROI bar chart
Per-station ROI ranking — surface which BTP stations have highest enforcement effectiveness
Redeployment recommendation: 'BTP043 has 38% ROI in Upparpet. Redeploy to Koramangala where ROI is 82%'

F8: Chronic Zone Classification
NICE TO HAVE

Purpose
Separate zones into CHRONIC (structural problem — appears in top-20 hotspots every week), EPISODIC (event-driven), and RANDOM (low frequency). Different interventions are required for each.

CHRONIC Zone
Appears in top-20 hotspots across 80%+ of weekly time windows. Needs infrastructure fix, not just enforcement.
EPISODIC Zone
Spikes on weekends, evenings, or event days. Needs scheduled patrol deployment.
RANDOM Zone
Low frequency, low severity. Monitor only.

Why Judges Care
Traffic Police leadership understands the difference between a zone that needs a tow truck and one that needs a road redesign. Presenting this classification shows operational maturity beyond what any analytics dashboard provides.

Core spatial input for H3 binning and HDBSCAN clustering
vehicle_type
Mapped to VehicleMass (M) in AOI formula. 7-tier scale.
created_date
Violation start time. Used for temporal features (hour, day, cyclic encoding).
closed_date
Violation end time. Δt = closed_date - created_date (obstruction duration).
junction_name
Binary junction flag. JunctionMultiplier = 2.0 if not null.
validation_status
Enforcement ROI computation. approved/rejected classification per zone.
center_code
Police station jurisdiction mapping for patrol optimization.
vehicle_number
Repeat offender intelligence. Frequency and multi-location tracking.
violation_type
Violation severity weighting. Arrays parsed into individual violation flags.

7.2 Handling Null closed_date
Approximately 15-25% of records will have null closed_date (violation still open or record incomplete). Strategy:
Compute rolling mean obstruction duration per H3 cell across completed records
Impute null closed_date as created_date + cell_mean_duration
Flag imputed records with is_duration_imputed = True
Evaluate model separately on imputed vs non-imputed records to check for data leakage

7.3 Feature Engineering Pipeline
# Cyclic time encoding
df['hour_sin'] = np.sin(2 * np.pi * df['hour'] / 24)
df['hour_cos'] = np.cos(2 * np.pi * df['hour'] / 24)
df['day_sin']  = np.sin(2 * np.pi * df['day_of_week'] / 7)
df['day_cos']  = np.cos(2 * np.pi * df['day_of_week'] / 7)

# Vehicle mass mapping
MASS_MAP = {'TANKER':5.0,'BUS':4.5,'CAR':3.0,'PASSENGER':3.0,
            'TAXI':2.5,'MOTOR CYCLE':1.5,'SCOOTER':1.0}
df['vehicle_mass'] = df['vehicle_type'].map(MASS_MAP).fillna(1.0)

# H3 binning
import h3
df['h3_cell'] = df.apply(lambda r: h3.geo_to_h3(r.latitude, r.longitude, 9), axis=1)

# Obstruction duration
df['duration_mins'] = (df['closed_date'] - df['created_date']).dt.total_seconds()/60

# Junction flag
df['junction_flag'] = df['junction_name'].notna().astype(int)

# Local density (rolling 3h window per H3 cell)
df = df.sort_values('created_date')
df['local_density'] = df.groupby('h3_cell')['id'].transform(
    lambda x: x.rolling('3h', on=df.loc[x.index,'created_date']).count())

# AOI computation
df['AOI_raw'] = df['vehicle_mass'] * df['duration_mins'] * df['local_density'] * df['junction_mult']
df['AOI'] = (df['AOI_raw'] / df['AOI_raw'].max() * 100).clip(0, 100)
